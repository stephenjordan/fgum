# fgum

Software to reproduce the results in "Optimization Using Locally-Quantum Decoders" [arXiv:2604.24633].

## Build

This curated archive builds with GNU Make and a C++20 compiler. 
```sh
make
```

This builds the following executables in `bin/`:

- `test_xortools`
- `full_anneal`
- `random_regular`
- `density`

To run the bundled bitmatrix/xortools test suite and a small density-evolution smoke test:

```sh
make check
```

To remove built binaries and generated object/dependency files:

```sh
make clean
```

## Simulated Annealing

The values in our table for simulated annealing are determined empirically by generating max-xorsat instances from Gallager's ensemble (with uniformly random v) and then running simulated annealing. Our C++ implementation of simulated annealing is called full_anneal. The commands to generate the problem instances and run the anneals are contained in sa_data/sa_jobs.txt. This file is generated by generate_sa_jobs.py. It might take a long time to run all of the anneals on a single machine. We ran them in parallel on a cluster. The results from the anneals that we ran on the cluster are contained in summary.txt, which we generated by the command "grep satisfied log* > summary.txt". Simulated annealing is a randomized algorithm and can benefit from running several trials. Therefore, for each instance we ran four separate anneals and took the best result. The script to do this and tabulate the results is sa_data/generate_sa_table.py.

## DQI + BP

One can run BP to find out its performance at any given block size empirically. However, one can instead obtain the precise asymptotic performance numbers using a technique called density evolution, which is what we did. Our C++ implementation is called density. The series of density evolution calculations can be run using bp_data/de.sh. These are quick enough that a cluster is not needed, so the script de.sh just runs these in parallel. The results are in bp_data/de_results.txt. The script bp_data/generate_bp_table.py parses these results and then uses them to determine the performance of DQI+BP via the semicircle law.

## Regev + FGUM

The calculation of the performance of Regev+FGUM is done using the equations from the manuscript. The equation for e_max is a univariate nonlinear equation that needs to be solved by numerical rootfinding. The script fgum_data/generate_fgum_table.py does this, and then substitutes the results into the necessary formulas to calculate the performance of Regev+FGUM.

## QAOA

The performance of QAOA is the most difficult and computationally intensive calculation reported in Table 1. We have implemented to methods do do this, which are available at:

https://github.com/jpmorganchase/QOKit/tree/add-max-k-xor-sat/qokit/max_k_xor_sat

and

https://github.com/johnazariah/qaoa-xorsat

A stripped-down code that only verifies the energies implied by the optimized angles without also optimizing the angles is also available at:

https://github.com/johnazariah/qaoa-verifier

The optimized angles are provided here in qaoa_data/qaoa_angles_for_verifier.csv. Generating these out to p=16 required extensive computations on modern GPUs.
